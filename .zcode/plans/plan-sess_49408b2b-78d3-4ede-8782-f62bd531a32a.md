## 部署计划:把 GitHub 最新代码部署到 121.89.81.236

### 背景
- 本地仓库落后 origin/main 2 个提交(本地 `c161340` → 远程 `e2eb730`)
- 服务器是**正在运行的生产服务** `chbzg-zego-rtc-test.service`,运行 6月30日构建的 JAR
- 服务器有成熟发布流程:`releases/<name>/` + `current` 软链 + `backups/` + systemd + `shared/.env`(密钥,不读取)
- 本机有完整构建链:Java 8 + Maven 3.9.9 + Node 22 + npm 10.9
- 服务器没 mvn/node,所以**本地构建后上传**

### 关键发现(需你知晓)
1. `scripts/package-test-release.sh` 引用了不存在的 `测试服务器部署方案.md`,直接跑会失败 → 我跳过这个脚本,手动组装发布包
2. 本地 `target/rtc-backend-1.0.0.jar` 是 7月3日旧产物,会用新代码重新构建
3. 前端构建输出到 `src/main/resources/static/`,会被打进 JAR
4. 录制 SDK 的 `libzegoliveroomrecorder.so` 是 Linux x64 native 库,本地不能运行验证,但可以打包上传

### 执行步骤

**第 1 步:同步代码到 GitHub 最新**
```
git fetch origin
git status           # 确认工作区干净
git merge --ff-only origin/main   # 快进合并到 e2eb730
```
(本地与远程是快进关系,1 ahead/2 behind 中的 1 ahead 是本地未推送的 `c161340`,实际是历史基底,可快进)

**第 2 步:本地构建前端**
```
cd front
npm install              # 安装依赖(有 node_modules 但版本可能变,重装稳妥)
npm run build            # 产物写入 ../src/main/resources/static/
cd ..
```

**第 3 步:构建 ZEGO 录制子程序**
```
./scripts/build-zego-recorder.sh   # javac --release 8,产物 tools/zego-recorder/build/zego-local-recorder.jar
```

**第 4 步:Maven 打包后端**
```
mvn clean package -DskipTests      # 产物 target/rtc-backend-1.0.0.jar
```

**第 5 步:本地组装发布目录**
按 `package-test-release.sh` 的结构(去掉不存在的 md 文件),在本地 `build/releases/chbzg-zego-rtc-test-deploy-<时间戳>/` 组装:
- `target/rtc-backend-1.0.0.jar`
- `scripts/run-zego-recorder.sh`
- `tools/zego-recorder/build/zego-local-recorder.jar`
- `third_party/zego-local-recording/libs/ZegoLiveRoom.jar` + `libzegoliveroomrecorder.so`
- `db/rtc_mysql.sql`
然后 `tar -czf` 打包 + `sha256sum` 生成校验

**第 6 步:上传到服务器**
用 paramiko(SFTP)上传 tar.gz 到 `/opt/chbzg-zego-rtc-test/`
- 上传前先检查服务器磁盘空间
- 服务器上 `tar -xzf` 解到 `releases/chbzg-zego-rtc-test-deploy-<时间戳>/`

**第 7 步:按现有流程切换发布(关键操作,会短暂影响线上)**
1. 备份当前:`cp -a current` 的 target 到 `backups/<时间戳>/`(保留旧 JAR 以便回滚)
2. 复制新 JAR 到新 release 的 `target/`
3. 更新 `current` 软链指向新 release
4. **不动 `shared/.env`**(密钥配置保持原样)
5. `systemctl restart chbzg-zego-rtc-test.service`
6. `systemctl status` + `tail app.log` 确认启动成功、端口 9003 监听

**第 8 步:验证**
- `systemctl status chbzg-zego-rtc-test` 显示 active running
- `ss -tlnp | grep 9003` 端口监听
- `curl http://127.0.0.1:9003/api/rtc/session` 或健康检查接口返回正常
- `tail -50 logs/app.log` 无异常堆栈

### 回滚方案(若启动失败)
```
ln -sfn /opt/chbzg-zego-rtc-test/releases/<旧release> /opt/chbzg-zego-rtc-test/current
systemctl restart chbzg-zego-rtc-test.service
```
旧 JAR 在 `backups/` 和旧 `releases/` 里都还在。

- 不删除任何 `releases/`、`backups/` 历史目录
- 不动 nginx、SRS、jitsi 配置
- 不向 GitHub push 任何东西(只 fetch/merge)
- 不在服务器装任何软件
