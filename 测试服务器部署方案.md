## 4. MySQL 建库与建表

实际测试部署采用隔离 Docker MySQL，不使用现有宝塔 MySQL：

```text
container: chbzg-zego-mysql
image: mysql:8.0
listen: 127.0.0.1:13306
database: chbzg_zego_rtc
volume: chbzg-zego-mysql-data
```

数据库凭据仅保存在服务器受限文件和 shared `.env`，文档不记录明文。

先由 MySQL 管理员创建独立库和最小权限用户：

```sql
CREATE DATABASE chbzg_zego_rtc
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

CREATE USER 'chbzg_zego_rtc'@'127.0.0.1'
  IDENTIFIED BY '使用独立强密码';

GRANT SELECT, INSERT, UPDATE, DELETE
  ON chbzg_zego_rtc.*
  TO 'chbzg_zego_rtc'@'127.0.0.1';

FLUSH PRIVILEGES;
```

再使用管理员账号执行：

```bash
mysql chbzg_zego_rtc < db/rtc_mysql.sql
```

只读验证：

```sql
SHOW TABLES;
SHOW CREATE TABLE rtc_room_state;
SHOW CREATE TABLE rtc_audit_log;
```

当前服务器 root 的无密码 socket 登录被拒绝，正式执行前需要提供 MySQL 管理员凭据，或在宝塔面板中执行建库和建表。

## 5. 环境变量

在 `/opt/chbzg-zego-rtc-test/shared/.env` 增加：

```bash
RTC_PERSISTENCE_ENABLED=true
RTC_PERSISTENCE_FAIL_FAST=true
RTC_MYSQL_JDBC_URL=jdbc:mysql://127.0.0.1:3306/chbzg_zego_rtc?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&useSSL=false
RTC_MYSQL_USERNAME=chbzg_zego_rtc
RTC_MYSQL_PASSWORD=独立数据库密码

RECORDING_ENABLED=true
RECORDING_WORK_DIR=/data/chbzg-zego-rtc/recordings
RECORDING_COMMAND_TEMPLATE=/opt/chbzg-zego-rtc-test/current/scripts/run-zego-recorder.sh --app-id {appID} --room-id {roomID} --user-id {userID} --token {token} --output {outputFile} --mode mix --log-dir /data/chbzg-zego-rtc/zego-logs
RECORDING_MAX_SECONDS=600
RECORDING_UPLOAD_MAX_ATTEMPTS=3
RECORDING_UPLOAD_RETRY_DELAY_MS=15000
RECORDING_DELETE_LOCAL_AFTER_UPLOAD=true
RECORDING_FAILED_FILE_RETENTION_HOURS=72
RECORDING_MIN_FREE_DISK_MB=2048
RECORDING_ADMIN_TOKEN=后台口令，服务器 .env 保存，不写入文档
OSS_PLAY_URL_EXPIRE_SECONDS=7200
```

`RECORDING_MAX_SECONDS=600` 对应当前业务确认的 10 分钟自动挂断和录制上限，本阶段不得擅自延长或取消。

现有 ZEGO、OSS 密钥继续放在 `.env`，不得写入发布包或文档。

## 6. 发布步骤

1. 本地执行测试、Web 构建和发布包构建。
2. 上传 release 到 `/opt/chbzg-zego-rtc-test/releases/`。
3. 备份当前 JAR、`.env`、systemd unit，并记录 SHA-256。
4. MySQL 建库建表，先验证 Java 使用的账号可连接。
5. 将 `.env` 移到 `shared/.env`，systemd 改为读取 shared 配置。
6. 首次把 `current` 目录转换为指向 release 的软链。
7. `systemctl daemon-reload`，重启 `chbzg-zego-rtc-test`。
8. 观察 2 分钟日志，确认 MySQL 初始化和 9003 监听成功。

建议执行命令：

```bash
cd /opt/chbzg-zego-rtc-test
sha256sum -c chbzg-zego-rtc-test-final-20260611-1012.tar.gz.sha256
tar -xzf chbzg-zego-rtc-test-final-20260611-1012.tar.gz -C releases/

sudo systemctl stop chbzg-zego-rtc-test
sudo ln -sfn \
  /opt/chbzg-zego-rtc-test/releases/chbzg-zego-rtc-test-final-20260611-1012 \
  /opt/chbzg-zego-rtc-test/current
sudo systemctl daemon-reload
sudo systemctl start chbzg-zego-rtc-test
sudo systemctl status chbzg-zego-rtc-test --no-pager
```

首次改造目录时，必须先将旧 `current` 实体目录改名备份，再创建软链，不能直接覆盖：

```bash
mv /opt/chbzg-zego-rtc-test/current \
   /opt/chbzg-zego-rtc-test/releases/pre-zego-lease-backup
ln -s /opt/chbzg-zego-rtc-test/releases/chbzg-zego-rtc-test-final-20260611-1012 \
      /opt/chbzg-zego-rtc-test/current
```

健康验证：

```bash
curl -fsS https://zego-test.chbzg.com.cn/
curl -fsS 'https://zego-test.chbzg.com.cn/api/rtc/status?roomID=deploy-check'
systemctl is-active chbzg-zego-rtc-test
```

## 7. MySQL 持久化联调

使用唯一测试房间依次调用：

1. 医生 `/api/rtc/session`
2. 患者 `/api/rtc/session`
3. 双方 `/api/rtc/heartbeat`
4. 患者 `/api/rtc/leave`
5. 医生 `/api/rtc/end`
6. 房间结束后再次申请 session，应返回 `当前订单已结束`

数据库验证：

```sql
SELECT room_id, status, doctor_user_id, peer_user_id,
       last_event, last_heartbeat_at, ended_at
FROM rtc_room_state
WHERE room_id = '指定测试房间';

SELECT action, result, room_id, user_id, role, message, created_at
FROM rtc_audit_log
WHERE room_id = '指定测试房间'
ORDER BY id;
```

## 8. 录制落盘与 OSS 联调

1. Web 医生以 `record=true` 进入测试房间。
2. 小程序进入同一房间并正常推流。
3. 确认录制 Java 进程存在，日志出现登录成功和发现远端流。
4. 医生挂断后确认进程退出。
5. 验证 MP4 非空，并使用 `ffprobe` 验证音视频轨道和时长。
6. 确认 OSS object 已生成，URL 返回 200 且可播放。
7. 人工中断一次 OSS 网络，验证三次重试和错误状态。

未满足以下条件时不能标记录制通过：

- MP4 大于 0 字节
- `ffprobe` 能读取
- 至少包含预期的视频或音频轨
- OSS 文件大小与本地一致
- 浏览器能够播放 OSS 地址；私有桶需改为签名 URL

## 9. 回滚方案

- 应用回滚：将 `current` 软链切回上一个 release，重启服务。
- 配置回滚：恢复 `shared/.env` 的时间戳备份。
- systemd 回滚：恢复 unit 备份并执行 `daemon-reload`。
- 数据库表为新增表，不影响旧网易业务；应用回滚时保留表，不执行 DROP。
- 若新服务启动失败，Nginx 不改，9003 恢复旧 release 后即可恢复测试域名。
